# CBTRN02C — Business-Logic Specification

*Source: `aws-samples/aws-mainframe-modernization-carddemo`, `app/cbl/CBTRN02C.cbl`. The COBOL source in `legacy/CBTRN02C.cbl` is copied without business-logic changes.*

## Purpose

`CBTRN02C` posts the daily transaction feed. For each valid transaction it creates a transaction-master record, updates the account balance and current-cycle credit/debit totals, and creates or updates the balance for the `(account, transaction type, transaction category)` key. Invalid transactions are written to the daily-reject file.

## Inputs

| DD name | Organization | Record and key | Layout |
|---|---|---|---|
| `DALYTRAN` | Sequential | 350-byte daily transaction | `CVTRA06Y` |
| `XREFFILE` | Indexed, random | 50-byte card cross-reference; key `XREF-CARD-NUM` | `CVACT03Y` |
| `ACCTFILE` | Indexed, random, I-O | 300-byte account; key `ACCT-ID` | `CVACT01Y` |
| `TCATBALF` | Indexed, random, I-O | 50-byte category balance; key `(TRANCAT-ACCT-ID, TRANCAT-TYPE-CD, TRANCAT-CD)` | `CVTRA01Y` |

### Daily transaction (`CVTRA06Y`, 350 bytes)

| Field | Picture | Notes |
|---|---|---|
| `DALYTRAN-ID` | `X(16)` | Transaction-master key |
| `DALYTRAN-TYPE-CD` | `X(02)` | Credit/purchase or debit/return classification |
| `DALYTRAN-CAT-CD` | `9(04)` | Transaction category |
| `DALYTRAN-SOURCE` | `X(10)` | Source channel |
| `DALYTRAN-DESC` | `X(100)` | Description |
| `DALYTRAN-AMT` | `S9(09)V99` | Signed zoned decimal; integer cents in the modern service |
| `DALYTRAN-MERCHANT-ID` | `9(09)` | Merchant identifier |
| `DALYTRAN-MERCHANT-NAME` | `X(50)` | Merchant name |
| `DALYTRAN-MERCHANT-CITY` | `X(50)` | Merchant city |
| `DALYTRAN-MERCHANT-ZIP` | `X(10)` | Merchant ZIP/postal code |
| `DALYTRAN-CARD-NUM` | `X(16)` | Lookup key into `XREFFILE` |
| `DALYTRAN-ORIG-TS` | `X(26)` | Original timestamp; first ten bytes are `YYYY-MM-DD` |
| `DALYTRAN-PROC-TS` | `X(26)` | Replaced with the posting timestamp |
| filler | `X(20)` | Spaces in generated transaction records |

### Card cross-reference (`CVACT03Y`, 50 bytes)

| Field | Picture | Notes |
|---|---|---|
| `XREF-CARD-NUM` | `X(16)` | Indexed key |
| `XREF-CUST-ID` | `9(09)` | Customer ID |
| `XREF-ACCT-ID` | `9(11)` | Account lookup key |
| filler | `X(14)` | The upstream ASCII extract omits trailing spaces; the loader pads them |

### Account fields used (`CVACT01Y`, 300 bytes)

| Field | Picture | Behavior |
|---|---|---|
| `ACCT-ID` | `9(11)` | Indexed key |
| `ACCT-CURR-BAL` | `S9(10)V99` | Add every accepted transaction amount |
| `ACCT-CREDIT-LIMIT` | `S9(10)V99` | Compared with cycle credit minus cycle debit plus the new amount |
| `ACCT-EXPIRAION-DATE` | `X(10)` | Lexical `YYYY-MM-DD` comparison with transaction date |
| `ACCT-CURR-CYC-CREDIT` | `S9(10)V99` | Add amounts greater than or equal to zero |
| `ACCT-CURR-CYC-DEBIT` | `S9(10)V99` | Add negative amounts without changing their sign |

### Transaction-category balance (`CVTRA01Y`, 50 bytes)

| Field | Picture | Notes |
|---|---|---|
| `TRANCAT-ACCT-ID` | `9(11)` | Composite key part |
| `TRANCAT-TYPE-CD` | `X(02)` | Composite key part |
| `TRANCAT-CD` | `9(04)` | Composite key part |
| `TRAN-CAT-BAL` | `S9(09)V99` | Existing balance plus the accepted amount |
| filler | `X(22)` | Preserved/padded |

## Outputs

| DD/output | Organization | Behavior |
|---|---|---|
| `TRANFILE` | Indexed, output | One 350-byte `CVTRA05Y` record per accepted transaction, with a generated processing timestamp |
| `DALYREJS` | Sequential, output | Original 350-byte transaction plus a 4-digit reason and 76-byte description |
| `ACCTFILE` | Indexed, rewritten | Updated current and current-cycle balances |
| `TCATBALF` | Indexed, rewritten/created | Updated or newly created category-balance records |
| stdout | Text | Start/end markers, missing-category creation notices, and zero-padded processed/rejected counts |

The parity baseline is `legacy/out/cbtrn02c.golden.txt`, regenerated by `legacy/run_cbtrn02c.sh`. The fixture has three accepted transactions and exercises both the existing-category and create-category paths.

## Validation and ordering

1. Look up the card number in `XREFFILE`; failure reason `0100` means invalid card.
2. Look up the referenced account; failure reason `0101` means missing account.
3. Compute `current-cycle-credit - current-cycle-debit + transaction amount`. If it exceeds the credit limit, reason `0102` is set.
4. Compare account expiration date with the first ten bytes of the original timestamp. An expired account sets reason `0103`; because this check follows the limit check, it can replace reason `0102`.
5. Accepted records are processed in daily-file order. There is no sort or control break.
6. Category balance is updated before the account record, then the transaction-master record is written.

## Parity-critical numeric semantics

- `S9(n)V99` values are fixed-width zoned decimal with an implied decimal point. The final byte carries the sign: `{`/`A`–`I` for positive final digits 0–9 and `}`/`J`–`R` for negative final digits 0–9.
- The TypeScript implementation stores monetary values as integer cents, avoiding binary floating-point rounding.
- Negative transactions are added directly to `ACCT-CURR-CYC-DEBIT`; the legacy program does not take the absolute value.
- Record widths and zero padding are preserved when account, category-balance, transaction, and reject records are rendered.
- `FUNCTION CURRENT-DATE` supplies the processing timestamp in `YYYY-MM-DD-HH.MM.SS.hh0000` form. It does not affect the stdout parity slice.

## Off-platform dependencies

- `CEE3ABD` is the only external `CALL`. The existing `legacy/CEE3ABD.cob` stub surfaces abnormal termination off-mainframe.
- `legacy/CBTRN02CLOAD.cob` replaces IDCAMS staging by loading the deterministic account, card-xref, and category-balance extracts into GnuCOBOL indexed files.
- `legacy/run_cbtrn02c.sh` maps the six DD names to local paths and converts the line-oriented daily fixture to the fixed-record sequential form expected by the unmodified program.

## Modernization mapping

`modern/src/cbtrn02c.ts` parses the same fixed-width records, applies the validation/posting order above, renders updated records with canonical zoned-decimal bytes, and emits the same stdout lines. `modern/test/cbtrn02c.parity.test.ts` asserts stdout parity byte-for-byte and separately verifies overpunch codecs and posted balances.
